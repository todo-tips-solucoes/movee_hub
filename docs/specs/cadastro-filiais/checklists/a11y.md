# A11y Checklist: Cadastro de Filiais

**Purpose**: Valida a qualidade dos requisitos de acessibilidade do formulário "Cadastrar filial" — contraste, focus rings, aria-live para erros, autocomplete semântico e touch targets.
**Created**: 2026-06-09
**Feature**: [spec.md](../spec.md)

## Contraste e Legibilidade

- [ ] CHK001 - Há requisito de contraste mínimo (WCAG AA 4.5:1 para texto normal) especificado para o formulário? [Completude, Gap — a11y não mencionada na spec] {humano}
  > Spec e contrato não mencionam requisitos de contraste. WCAG 2.1 AA é o baseline padrão para produtos B2B, mas deve ser declarado explicitamente para que a implementação e review possam verificar. Decisão: adotar WCAG AA como requisito explícito?

- [ ] CHK002 - Há requisito de contraste para estados de foco (focus rings visíveis com razão >= 3:1)? [Completude, Gap — a11y não mencionada na spec] {humano}
  > Nenhum requisito de focus ring está na spec. Para usuários de teclado, focus rings invisíveis tornam o formulário inacessível. Decisão: especificar razão de contraste mínima para o indicador de foco?

- [ ] CHK003 - Há requisito de contraste para mensagens de erro (texto vermelho sobre fundo claro deve atingir 4.5:1)? [Completude, Gap] {humano}
  > FR-008 especifica erro abaixo do campo, mas não o contraste. Vermelho puro (#FF0000) sobre branco falha WCAG AA para texto pequeno. Decisão: cor de erro deve ser especificada com valor de contraste verificável?

## Focus Management

- [x] CHK004 - O requisito de foco automático no primeiro campo inválido ao submeter está especificado? [Completude, Spec §FR-008] {auto}
  > FR-008: "foco automático no primeiro campo inválido ao submeter." Requisito presente e testável.

- [ ] CHK005 - Há requisito de ordem de foco (tab order) compatível com a ordem visual dos campos? [Completude, Gap] {humano}
  > Spec não menciona tab order. Para formulários com múltiplos campos, tab order incorreto (ex: campos fiscais opcionais fora de sequência visual) quebra a experiência de teclado. Decisão: especificar que tab order segue a ordem visual de cima para baixo?

- [ ] CHK006 - O comportamento de foco ao abrir o formulário/modal está especificado (foco vai para o primeiro campo ou para o heading)? [Clareza, Gap] {humano}
  > Spec não especifica onde o foco cai ao abrir o formulário. Para usuários de teclado e leitores de tela, foco em posição inesperada torna o formulário difícil de navegar. Decisão: foco no primeiro campo (nome_empresa) ou no título do formulário ao abrir?

## ARIA e Semântica

- [ ] CHK007 - Há requisito de `aria-live` ou `role="alert"` para anúncio das mensagens de erro aos leitores de tela? [Completude, Gap — a11y não mencionada na spec] {humano}
  > FR-008 especifica erro abaixo do campo visualmente, mas sem `aria-live="polite"` ou `role="alert"`, leitores de tela não anunciam os erros dinamicamente. Usuários cegos submeteriam o formulário sem feedback. Decisão: especificar que erros usam `aria-live` ou `role="alert"`?

- [ ] CHK008 - Há requisito de `aria-describedby` ou `aria-errormessage` associando cada mensagem de erro ao campo correspondente? [Completude, Gap] {humano}
  > Visualmente o erro fica abaixo do campo, mas a associação semântica (campo + mensagem) não está especificada. Sem `aria-describedby`, leitores de tela não relacionam o erro ao input. Decisão: especificar associação programática entre campo e sua mensagem de erro?

- [ ] CHK009 - Há requisito de `aria-invalid="true"` nos campos com erro? [Completude, Gap] {humano}
  > `aria-invalid` comunica o estado de erro ao leitor de tela independentemente do CSS. Não mencionado na spec. Decisão: incluir como requisito de implementação?

- [ ] CHK010 - O feedback de carregamento (loading) tem requisito de anúncio para leitores de tela? [Completude, Gap] {humano}
  > FR-009 especifica "feedback de carregamento" visualmente, mas não como usuários de teclado/leitores de tela saberão que a requisição está em andamento. Decisão: spinner com `aria-label="Salvando..."` ou `aria-live` com status de progresso?

## Autocomplete e Semântica de Input

- [ ] CHK011 - Há requisito de atributos `autocomplete` semânticos nos campos do formulário? [Completude, Gap] {humano}
  > Spec não menciona `autocomplete`. Para campos como `email` (`autocomplete="email"`) e `nome_empresa` (`autocomplete="organization"`), o atributo correto melhora preenchimento automático e acessibilidade. Decisão: especificar `autocomplete` para os campos principais?

- [ ] CHK012 - Há requisito de atributo `type` semântico para os inputs (type="email", type="password", type="tel" para CEP/CNPJ)? [Completude, Gap] {humano}
  > Spec descreve campos funcionalmente, mas não especifica os `type` HTML semânticos. `type="email"` ativa teclado correto em mobile; `type="password"` esconde a senha. Decisão: especificar `type` para cada campo, especialmente `email` e `senha`?

- [ ] CHK013 - O campo de CNPJ tem requisito de `inputmode` ou máscara especificado para teclado numérico em mobile? [Completude, Gap] {humano}
  > Spec §Edge Cases menciona que a UI "pode aceitar ou limpar a máscara antes de enviar", mas não especifica se o campo deve mostrar apenas dígitos, usar máscara visual, ou ter `inputmode="numeric"` para abrir teclado numérico em mobile. Decisão: comportamento do campo CNPJ no mobile?

## Touch Targets

- [ ] CHK014 - Há requisito de tamanho mínimo de touch target (>= 44x44 px) para os botões do formulário? [Completude, Gap — a11y não mencionada na spec] {humano}
  > WCAG 2.5.5 (AAA) e WCAG 2.5.8 (AA em 2.2) recomendam 44px mínimo. Spec não menciona tamanhos de touch target. Para o botão "Cadastrar" e botões de ação na lista de filiais, este requisito é especialmente relevante em dispositivos móveis. Decisão: adotar 44px mínimo como requisito?

- [ ] CHK015 - Há requisito de espaçamento adequado entre o botão de desvincular e outros elementos interativos (evitar cliques acidentais)? [Completude, Gap] {humano}
  > Botão de desvincular na lista de filiais não tem requisito de espaçamento. Em listas densas, cliques acidentais em "desvincular" têm impacto alto. Decisão: especificar espaçamento mínimo entre ações destrutivas e outras ações na lista?

## Responsividade e Formulário Mobile

- [ ] CHK016 - Há requisito de comportamento do formulário em telas pequenas (breakpoints, scroll, layout)? [Completude, Gap] {humano}
  > Spec não menciona responsividade. Um formulário de 9 campos sem requisito de layout mobile pode resultar em campos cortados ou sobrepostos. Decisão: especificar breakpoints ou referência ao design system existente do projeto?

## Notes

- Items `{auto}` resolvidos com citação de evidência
- **Todos os 15 items de a11y são `{humano}`** — a spec não menciona acessibilidade em nenhuma seção
